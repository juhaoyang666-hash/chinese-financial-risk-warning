# 中文金融舆情风险智能预警系统：故事背景与技术路线

作者：10225001426 杨巨豪

## 1. 项目背景：从 NLP 大作业到风控算法项目

本项目最初来源于 NLP 课程大作业，选题为“中文金融负面主体风险识别”。原始任务是：给定一段金融新闻、公告或社交媒体文本，判断其中是否存在负面金融风险信息，并抽取负面信息指向的金融主体。

这个任务看起来是一个文本分类和实体抽取任务，但它背后的业务场景非常接近真实金融风控：互金平台、信贷机构、资管机构、支付公司和消费金融公司每天都会面对大量新闻、投诉、社媒爆料、监管通报和司法信息。风控团队不仅关心一条文本是不是负面，更关心三个问题：

1. 负面风险具体指向哪个主体。
2. 风险属于什么类型，严重程度如何。
3. 这条风险信息是否需要进入人工审核或主体画像。

因此，项目没有停留在“情感分类”层面，而是逐步升级为一个“中文金融舆情风险智能预警系统”。系统目标是从中文金融文本中识别风险主体、抽取风险事件、生成风险评分，并通过 API、数据库和审核台形成一个可以演示、可以复现实验、也适合写进简历的风控 NLP 项目。

## 2. 业务痛点与项目定位

普通文本情感分类通常只输出“正面/负面/中性”，但金融风控场景对模型有更高要求：

- 金融文本往往包含多个主体，例如平台、监管机构、银行、借款人、出借人和媒体。
- 风险信息常常以社交媒体口语、投诉文本、标题党新闻或司法通报形式出现，噪声较大。
- 风控系统更重视召回高风险线索，但也需要控制误报，避免审核队列过载。
- 模型结果必须可解释，需要给出证据片段、风险类型和处置建议。
- 线上系统需要高吞吐初筛，复杂样本才交给大模型复核。

因此，本项目定位为：

> 面向互金、信贷、支付和资管场景，构建一个从金融新闻/社交媒体文本中识别风险主体、抽取风险事件、生成风险评分，并支持级联推理、实体画像和人工审核的中文金融舆情风险预警系统。

相比单纯的 NLP 分类实验，这个定位更贴近风控算法岗需要的能力组合：

- NLP/LLM：中文金融文本理解、实体归因、指令微调、JSON 结构化输出。
- 风控算法：风险分层、召回优先、KS/PSI、主体画像、审核队列。
- 工程落地：FastAPI、Streamlit、SQLite、可复现实验脚本、提交包打包。
- 研究规范：多模型对比、外部泛化、反事实鲁棒性、错误分析和漂移监控。

## 3. 数据与任务定义

### 3.1 主任务数据：FINNSP

主数据集来自 FinCUGE-Instruction 中的 FINNSP 任务。该任务对应金融负面消息及其主体判定，适合构建中文金融负面主体识别实验。

当前处理后的数据规模为：

| 数据集 | 样本数 | 正例 | 负例 |
|---|---:|---:|---:|
| Train | 4000 | 2221 | 1779 |
| Eval | 500 | 259 | 241 |

项目将原始 instruction 数据统一为 JSONL 格式：

```json
{
  "id": "train-0",
  "text": "金融新闻或社交媒体文本",
  "has_negative": true,
  "label": 1,
  "entities": ["小资钱包", "资易贷"],
  "target_json": "{\"has_negative\": true, \"entities\": [\"小资钱包\", \"资易贷\"]}"
}
```

无负面主体样本统一为：

```json
{
  "has_negative": false,
  "entities": []
}
```

### 3.2 标签归一化

原始 FINNSP 输出不是严格 JSON，存在多种自然语言表达，例如“有”“该文本中不包含负面金融实体”“文中包含的负面主体：小资钱包”等。项目实现了统一解析逻辑：

- 识别“不包含、没有、未发现、无负面”等无风险表达。
- 支持中文冒号、分号、顿号、逗号等分隔符。
- 将输出统一为 `has_negative` 和 `entities`。
- 对 LLM 非法 JSON 输出提供兜底解析。
- 支持新版 `risk_events` schema，同时兼容旧版 `entities` 输出。

这一步保证传统模型、编码器模型和大语言模型可以在同一评估口径下比较。

### 3.3 外部泛化数据

为了避免项目只像课程作业，额外加入 FLARE-zh-NSP 作为同任务外部二分类评估数据。当前外部评估样本数为 500 条，结果用于说明模型或规则在跨数据源场景下的泛化能力。

### 3.4 结构化风控数据

为了把项目从 NLP 扩展到完整风控算法场景，新增 OpenML/UCI 真实信用卡违约数据：

- 数据源：OpenML `default-of-credit-card-clients`
- 固定 ID：`data_id=42477`
- 样本数：30000
- 特征数：23
- 任务：预测信用卡客户下月是否违约
- 备用真实数据源：OpenML `credit-g`，`data_id=31`

该分支用于展示传统 tabular 风控建模能力，与文本舆情风险分数形成“文本风险 + 结构化风险”的扩展方向。

## 4. 总体系统架构

当前系统可以拆成六层：

```text
数据层
  FINNSP / FLARE-zh-NSP / OpenML Credit Default / 自建样例
  金融新闻、社媒文本、候选实体、结构化风控特征

模型层
  TF-IDF / RoBERTa / MacBERT / FinBERT2
  Qwen3-8B QLoRA
  LightGBM tabular risk model

策略层
  FinBERT2 高吞吐初筛
  Qwen3-8B QLoRA 复核
  实体候选过滤
  风险类型弱监督
  风险评分和审核动作

画像层
  risk_events
  entity_profiles
  review_queue
  prediction_logs
  drift_snapshots

服务层
  FastAPI /score
  FastAPI /batch_score
  FastAPI /entity/{name}
  FastAPI /review_queue
  FastAPI /metrics

展示与评估层
  Streamlit 审核台
  结果表、图表、错误分析
  外部泛化、反事实鲁棒性、漂移监控
```

系统核心思想是：用轻量模型做高吞吐初筛，用 Qwen3-8B QLoRA 做复杂样本复核和实体归因，再把输出转成风控系统可消费的风险事件、风险评分和主体画像。

## 5. 模型技术路线

### 5.1 传统 NLP 基线

第一组基线使用字符级 TF-IDF 表示文本，再训练线性分类器：

- TF-IDF + Logistic Regression
- TF-IDF + Linear SVM

这组模型训练快、资源消耗低、可解释性强，用于证明后续预训练模型和 LLM 的收益。

### 5.2 中文预训练编码器

第二组模型使用中文预训练编码器进行二分类：

- Chinese RoBERTa-wwm-ext
- Chinese MacBERT-base
- FinBERT2-large

编码器模型不负责主体生成，主要承担线上高吞吐初筛职责。当前最优编码器为 FinBERT2-large，验证集 F1 为 0.9638。

### 5.3 Qwen zero-shot / few-shot 探索

为了验证通用大模型在无训练条件下的能力，项目评估了 Qwen3-8B zero-shot 和 few-shot 设置。zero-shot 指只给任务说明和输出格式，不给标注示例；few-shot 指在提示词中加入少量训练样例及其标准 JSON 输出。复核后两组实验均关闭 thinking 并要求直接输出 JSON，格式可用性显著改善。zero-shot 验证集 F1 为 0.9150、Entity-F1 为 0.5526；few-shot 提升到 F1 0.9392、Entity-F1 0.6273。但二者 Precision 和 Entity-F1 仍明显低于 QLoRA 主模型，说明仅依靠提示词时仍存在风险主体范围扩大和实体误报问题。

最终报告保留 Qwen3-8B zero-shot / few-shot 作为“为什么需要指令微调”的对照，不把 Qwen3.5-9B QLoRA 纳入正式结果。

### 5.4 Qwen3-8B QLoRA 主模型

主模型为 Qwen3-8B QLoRA 指令微调，正式采用如下训练配置：

- Base model：Qwen3-8B
- LoRA adapter：低秩适配器参数，仅用于本地训练与评估，不随提交包提供
- 量化：4-bit NF4
- LoRA rank：16
- LoRA alpha：64
- Dropout：0.05
- Learning rate：2e-4
- Max sequence length：2048
- 训练轮数：3 epoch
- 训练方式：4 卡 `accelerate` 多进程训练

训练目标是让模型稳定输出 JSON：

```json
{"has_negative": true, "entities": ["主体A", "主体B"]}
```

QLoRA 的价值在于用较低显存成本完成领域适配，在 3090 服务器上能够完成 8B 级模型训练。

## 6. 风控系统升级路线

### 6.1 从 `entities` 升级到 `risk_events`

原始 NLP 任务只输出：

```json
{"has_negative": true, "entities": ["小资钱包"]}
```

系统升级后输出风险事件：

```json
{
  "has_negative": true,
  "risk_events": [
    {
      "entity": "小资钱包",
      "risk_type": "兑付/逾期风险",
      "severity": "high",
      "confidence": 0.91,
      "risk_score": 86.5,
      "risk_level": "high",
      "evidence": "到期就逾期，不回本金",
      "action": "人工复核"
    }
  ]
}
```

固定 7 类风险类型：

- 诈骗/非法集资
- 兑付/逾期风险
- 监管/处罚风险
- 经营异常
- 财务恶化
- 投诉纠纷
- 洗钱/涉黑等严重违法

风险等级固定为：

```text
low < medium < high < critical
```

需要注意的是，当前 `risk_type`、`severity`、`evidence` 和 `risk_score` 是基于 FINNSP 主体标签与关键词规则构造的风控扩展弱标签，不是人工标注的风险事件多分类任务。

### 6.2 风险评分

系统使用可解释的加权评分：

```text
risk_score = 0.35 * model_confidence
           + 0.25 * severity_score
           + 0.20 * source_reliability
           + 0.10 * entity_history_score
           + 0.10 * recency_score
```

其中：

- `model_confidence` 来自模型置信度或级联策略。
- `severity_score` 反映风险类型和严重程度。
- `source_reliability` 区分监管公告、警方通报、新闻媒体、社媒。
- `entity_history_score` 表示主体历史风险频次。
- `recency_score` 表示近期事件权重。

处置动作根据风险等级和置信度生成，例如“人工复核”“重点关注”“入库观察”等。

### 6.3 实体候选约束

大语言模型可能生成原文中不存在的主体。为降低主体幻觉，项目实现实体候选过滤：

- 优先使用原始标签实体。
- 从文本中抽取公司、平台、银行、金融等候选主体。
- LLM 输出后过滤不在候选集或不在原文中的主体。
- 统计实体在文率和幻觉实体率。

当前 Qwen3-8B QLoRA 预测实体总数为 441，其中 435 个能在原文匹配：

- 实体在文率：`98.64%`
- 幻觉实体率：`1.36%`

### 6.4 SQLite 风控库

系统使用 SQLite 作为轻量本地风控库，数据库位于：

```text
outputs/risk_system/risk_system.db
```

当前表设计包括：

- `risk_events`：风险事件明细。
- `entity_profiles`：主体画像。
- `review_queue`：待审核队列。
- `prediction_logs`：预测日志。
- `drift_snapshots`：漂移快照。

当前落库规模基于 FINNSP eval gold 主体和关键词弱标签构建，主要用于本地演示和主体画像样例，不等同于线上模型自动沉淀的生产库：

| 表 | 记录数 |
|---|---:|
| risk_events | 421 |
| entity_profiles | 173 |
| review_queue | 321 |
| prediction_logs | 0 |
| drift_snapshots | 2 |

### 6.5 FastAPI 与 Streamlit

服务端实现了真实模型加载和风险评分接口：

- `POST /score`：单条文本评分。
- `POST /batch_score`：批量文本评分并写入 SQLite。
- `GET /entity/{name}`：查询主体风险画像。
- `GET /review_queue`：查询待审核队列。
- `GET /metrics`：返回服务统计和漂移摘要。

Streamlit 审核台支持：

- 文本输入。
- 风险事件展示。
- 风险分数和证据片段展示。
- 主体画像查询。
- 审核队列查看。
- 当前模型指标展示。

这部分让项目从“离线实验”升级为“可演示系统”。

## 8. 核心实验结果

### 8.1 NLP 主任务结果

| 模型 | Accuracy | Precision | Recall | F1 | Macro-F1 | Entity-F1 |
|---|---:|---:|---:|---:|---:|---:|
| TF-IDF + Logistic Regression | 0.9300 | 0.9242 | 0.9421 | 0.9331 | 0.9299 | 0.0000 |
| TF-IDF + Linear SVM | 0.9500 | 0.9432 | 0.9614 | 0.9522 | 0.9499 | 0.0000 |
| Chinese RoBERTa | 0.9540 | 0.9538 | 0.9575 | 0.9557 | 0.9539 | 0.0000 |
| Chinese MacBERT | 0.9580 | 0.9542 | 0.9653 | 0.9597 | 0.9579 | 0.0000 |
| FinBERT2-large | 0.9620 | 0.9511 | 0.9768 | 0.9638 | 0.9619 | 0.0000 |
| Qwen3-8B QLoRA | 0.9720 | 0.9658 | 0.9807 | 0.9732 | 0.9719 | 0.8331 |

Qwen3-8B QLoRA 是主模型，二分类 F1 达到 `0.9732`，实体级 F1 达到 `0.8331`，非法 JSON 率为 `0.0`。最终配置相比初始 `r=16/alpha=32` 配置小幅提升了二分类和实体级指标，因此作为最终论文和系统默认方案。

### 8.2 级联风控离线模拟结果

级联策略为：FinBERT2-large 初筛，预测为正例的样本再交给 Qwen3-8B QLoRA 复核。该结果由已保存的 encoder/LLM 验证集预测文件离线组合得到，用于评估策略可行性和 LLM 调用率，不直接等同线上真实延迟收益。

| 策略 | Accuracy | F1 | Entity-F1 | LLM 调用率 |
|---|---:|---:|---:|---:|
| FinBERT2 only | 0.9620 | 0.9638 | 0.0000 | 0.0% |
| Qwen3-8B QLoRA all | 0.9720 | 0.9732 | 0.8331 | 100.0% |
| FinBERT2 -> Qwen3-8B QLoRA cascade | 0.9700 | 0.9710 | 0.8386 | 53.2% |

该结果说明：离线级联策略能在只调用约一半 LLM 的情况下，接近全量 LLM 的二分类性能，并保持较好的实体级效果，适合作为真实风控系统成本控制方案的原型依据。

### 8.3 可靠性分析

Qwen3-8B QLoRA 可靠性指标：

- Invalid JSON rate：`0.0`
- 实体在文率：`98.64%`
- 幻觉实体率：`1.36%`
- False positive：9
- False negative：5
- Entity mismatch：72
- 多主体样本数：158

主要错误集中在多主体归因和主体边界不一致。例如文本同时出现平台、关联公司、监管机构时，模型可能抽取过宽或漏掉某个风险主体。

### 8.5 反事实样本构造与 Qwen3 压力测试

构造 400 条反事实样本，并先用关键词规则 baseline 做初步检查。规则结果如下：

| 指标 | 结果 |
|---|---:|
| entity_switch_accuracy | 0.99 |
| negative_cue_sensitivity | 0.30 |
| distractor_resistance | 1.00 |

随后使用 Qwen3-8B QLoRA 主模型进行真实推理压力测试。该结果用于揭示模型边界，不作为主任务核心性能指标：

| 指标 | 结果 |
|---|---:|
| entity_switch_accuracy | 0.4500 |
| negative_cue_sensitivity | 0.0400 |
| distractor_resistance | 0.9500 |
| negation_robustness | 0.9400 |
| invalid_json_rate | 0.0000 |

结果说明，主模型对干扰主体和否定/澄清语境较稳，但对主体替换和负面触发词删除仍明显敏感。特别是 negative_cue_removed 指标较低，一方面反映模型倾向于保守预警，另一方面也说明自动构造样本仅删除少量显式负面词后，文本中可能仍残留“风险提示”“备案异常”等弱风险线索。该实验应作为压力测试，而不是人工标注鲁棒性基准。

## 11. 报告和答辩叙事建议

建议正式报告按下面逻辑讲：

1. 金融风控不是简单情感分类，关键是实体级风险归因。
2. FINNSP 提供了中文金融负面主体识别任务，适合作为 NLP 主实验。
3. 先建立传统基线和编码器基线，再用 Qwen3-8B QLoRA 做生成式实体抽取。
4. Qwen3-8B QLoRA 在 F1、Entity-F1、JSON 稳定性上表现最好。
5. 为了贴近真实风控系统，引入 FinBERT2 + Qwen3 的级联策略，降低 LLM 调用成本。
6. 将 `entities` 扩展为 `risk_events`，加入风险类型、证据、评分和处置动作。
7. 用 SQLite、FastAPI、Streamlit 搭出可演示的审核系统。
8. 用 FLARE-zh-NSP 外部二分类泛化、Qwen3 反事实压力测试和漂移监控说明项目不只停留在单一排行榜。
9. 用 OpenML 信用违约数据补充 tabular 风控分支，说明项目可以扩展到传统风控算法。

最终可以把项目概括为：

> 本项目构建了一个面向金融舆情风控的中文 NLP 预警系统，基于 FINNSP 数据完成多模型对比和 Qwen3-8B QLoRA 指令微调，主模型达到 F1 0.9732、Entity-F1 0.8331；进一步设计 FinBERT2 + Qwen3 离线级联模拟、实体候选约束、风险事件弱标签扩展、风险评分、主体画像和审核队列，并通过 FastAPI + Streamlit 实现可演示原型。同时补充 FLARE-zh-NSP 外部二分类泛化、Qwen3 反事实压力测试、漂移监控和 OpenML 真实信用违约 tabular 风控实验，使课程作业升级为更完整的风控算法项目原型。

## 12. 当前不足与后续优化

当前项目已经覆盖课程大作业和简历展示的核心能力，但仍有可优化空间：

- 多主体样本仍是主要错误来源，后续可引入金融 NER 或实体链接模型。
- 风险类型目前主要依赖弱监督关键词，可进一步构造人工标注小样本微调分类器。
- 负面触发词删除后的鲁棒性较弱，可以加入更多否定、澄清、反事实训练样本。
- SQLite 适合本地演示，若做生产级系统可升级到 PostgreSQL 或图数据库。
- Dashboard 目前偏审核展示，后续可加入审核反馈闭环和模型再训练数据沉淀。
- Tabular 分支当前独立训练，未来可设计主体匹配机制，把文本风险分数作为真实结构化特征接入。

整体来看，项目已经从“中文金融负面主体识别实验”升级为“中文金融舆情风险智能预警系统”的第一版完整原型。
