# 简历级风控项目升级路线

作者：10225001426 杨巨豪

## 1. 项目重新定位

当前项目作为 NLP 大作业已经合格：它完成了 FINNSP 数据处理、多个 baseline、Qwen3-8B QLoRA、统一 JSON 输出、实体 F1、错误分析和级联风控模拟。但如果目标是找风控算法实习，项目需要从“模型训练实验”升级为：

**中文金融舆情风险智能预警系统**

一句话定位：

> 面向互金、信贷、支付和资管场景，构建一个从金融新闻/社交媒体文本中识别风险主体、抽取风险事件、生成风险评分，并支持低成本级联推理和可解释分析的风控 NLP 系统。

这个定位比“负面主体识别”更适合简历，因为它同时覆盖：

- NLP/LLM：中文金融文本理解、实体归因、QLoRA 微调。
- 风控算法：风险分层、召回优先、误报控制、实体风险画像。
- 工程能力：FastAPI 服务、dashboard、监控、实验管理、可复现 pipeline。
- 业务理解：舆情早筛、人工审核分流、黑名单线索发现、贷前/贷中风险预警。

## 2. 参考项目给我们的启发

### 2.1 FinGPT

FinGPT 的核心价值不是单一模型，而是金融 LLM 生态：多任务 instruction tuning、金融数据、RAG、benchmark 和 open-source base model 对比。它使用多类金融 instruction 数据，例如 sentiment、relation extraction、headline、NER、QA 等。

对本项目的启发：

- 不要只做 FINNSP 单任务，要扩展为多任务金融风险理解。
- 可以加入 NER、关系抽取、风险事件分类，让项目看起来更像金融 NLP 平台。
- 可以保留 QLoRA，但更强调“低成本适配金融风险任务”。

参考：https://github.com/AI4Finance-Foundation/FinGPT

### 2.2 CALM

CALM 是信用与风险评估方向的 LLM 项目，覆盖数据构建、模型训练、benchmark 和 bias analysis。它把信用评分、欺诈检测、财务困境识别、理赔分析等任务统一为信用风险评估 instruction benchmark。

对本项目的启发：

- 风控项目不能只报 Accuracy/F1，还应该关注偏差、稳定性、错误代价。
- 可以把“金融负面主体识别”扩展成风险评估 instruction 数据集。
- 后续可加入 tabular 风控数据，把文本风险信号和传统信用风险模型融合。

参考：https://github.com/The-FinAI/CALM

### 2.3 OpenML/UCI 结构化信用风险数据

OpenML `default-of-credit-card-clients` 固定 `data_id=42477`，来源为 UCI 信用卡客户违约数据，包含 30,000 条客户额度、账单、还款状态和下月违约标签，适合作为结构化信用风险建模分支。

对本项目的启发：

- 简历项目可以加入一个 tabular 风控分支，展示你不仅会 NLP，也懂传统风控建模。
- 可以构造“文本风险信号 + 用户/机构结构化特征”的 hybrid risk score。
- 风控算法岗常见技能包括 AUC、KS、PSI、WOE/IV、分箱、评分卡、XGBoost/LightGBM。

备用真实公开数据源为 OpenML `credit-g`，固定 `data_id=31`。

### 2.4 FinEntity

FinEntity 关注金融实体级情感分类，强调多实体金融新闻里必须判断情绪到底指向哪个实体。

对本项目的启发：

- 我们当前做的是实体级负面风险识别，和 FinEntity 的“entity-level sentiment”方向天然接近。
- 论文和简历都应强调“实体归因”，这是区别普通文本分类的关键。
- 可以增加实体候选约束、实体在文率、实体幻觉率和 entity mismatch 分析。

参考：https://github.com/yixuantt/FinEntity

### 2.5 OpenNews

OpenNews 是实时金融新闻知识图谱与影响评分系统，包含多源新闻接入、实体抽取、主题聚类、影响评分、PostgreSQL/Neo4j 存储和 dashboard。

对本项目的启发：

- 简历项目最好有系统形态：输入新闻流，输出实体风险事件和风险分数。
- 可以做一个轻量版知识图谱：`Entity -> RiskEvent -> Evidence -> Source`。
- Dashboard 能显著提升展示效果，面试时比一张结果表更有说服力。

参考：https://github.com/IUnlimit/opennews

### 2.6 传统欺诈检测 / 风控 MLOps 项目

很多风控项目会包含 FastAPI、SHAP、SQL、监控、dashboard 和部署。典型思路是实时检测、降低误报、给审核人员 reason code。

对本项目的启发：

- LLM 输出应该转化为可解释 reason code，而不是只给 JSON。
- 应该提供 API 和可视化页面，模拟真实风控审核台。
- 需要监控：数据漂移、预测分布漂移、风险主体新增、LLM 格式失败率。

参考：https://github.com/sangramdedge/AI-Fraud-Detection-Transaction-Monitoring

## 4. 建议大改方向

### 4.1 从 `entities` 升级为 `risk_events`

当前 schema：

```json
{"has_negative": true, "entities": ["小资钱包", "资易贷"]}
```

建议升级为：

```json
{
  "has_negative": true,
  "risk_events": [
    {
      "entity": "小资钱包",
      "risk_type": "兑付/逾期风险",
      "severity": "high",
      "confidence": 0.91,
      "evidence": "到期就逾期，不回本金",
      "action": "人工复核"
    }
  ]
}
```

这样简历可以写“金融风险事件抽取”，不只是“情感分类”。

建议实现：

- 新增 `src/risk_nlp/risk_types.py`：关键词弱监督风险类型打标。
- 新增 `scripts/build_risk_events.py`：把已有 FINNSP 样本转成 risk event 弱标签。
- 新增 `risk_event_f1`、`risk_type_accuracy`、`severity_accuracy`。

### 4.2 实体候选约束

现在已经统计实体在文率和幻觉率，下一步应把“统计”变成“约束”。

建议实现：

- 从文本中解析 `candidate_entities`，包括原始“实体：A;B”、标签实体、公司/平台规则实体。
- prompt 中明确要求：只允许从候选实体中选择。
- 后处理时过滤不在候选集或不在原文中的实体。
- 对比实验：无约束 vs 候选约束。

可写进简历：

> 设计实体候选约束与后处理校验，将 LLM 幻觉主体率控制在低水平，并提升金融风险归因可信度。

### 4.3 风险评分系统

风控项目不能只有分类结果。建议设计一个 `risk_score`：

```text
risk_score = 0.35 * model_confidence
           + 0.25 * severity_score
           + 0.20 * source_reliability
           + 0.10 * entity_history_score
           + 0.10 * recency_score
```

字段解释：

- `model_confidence`：编码器概率或 LLM 置信度。
- `severity_score`：诈骗/非法集资 > 逾期兑付 > 投诉纠纷。
- `source_reliability`：监管公告/警方通报 > 新闻媒体 > 社媒。
- `entity_history_score`：同一主体历史风险事件越多越高。
- `recency_score`：近期事件权重更高。

建议实现：

- 新增 `src/risk_nlp/scoring.py`。
- 输出 `risk_level = low / medium / high / critical`。
- 生成 `outputs/risk_profiles/entity_profiles.json`。

可写进简历：

> 构建实体级风险评分模块，融合模型置信度、风险类型、来源可信度和历史事件频次，实现主体风险画像与审核优先级排序。

### 4.4 加入外部泛化数据

当前只有 FINNSP train/eval，像作业。要更像项目，建议至少加：

- `ChanceFocus/flare-zh-nsp`：同任务外部二分类测试。
- `OpenML default-of-credit-card-clients`：tabular 信用违约预测任务，作为结构化风险建模分支。
- 可选：FinEntity 或金融 NER 数据，用于实体抽取/实体级情感补充。

建议实现：

- `scripts/prepare_flare_nsp.py`
- `scripts/evaluate_external.py`
- `scripts/train_tabular_risk.py`

可写进简历：

> 在跨数据集评估中验证模型泛化能力，并扩展结构化违约/欺诈数据，实现文本风险信号与传统风控模型的联动。

### 4.5 增加反事实鲁棒性和压力测试

建议构造 100 条小型 counterfactual set：

- 替换风险主体名称。
- 删除负面触发词。
- 插入监管机构/媒体机构作为干扰主体。
- 将“谣传/澄清/否认”等否定语境加入负面文本。

指标：

- `entity_switch_accuracy`
- `negative_cue_sensitivity`
- `distractor_resistance`
- `negation_robustness`

可写进简历：

> 构造反事实鲁棒性测试集，验证模型在主体替换、负面词删除、干扰实体插入和否定语境下的风险归因稳定性。

### 4.6 API 服务和审核台

这是简历项目最容易“看起来像产品”的部分。

建议实现：

```text
app/
  main.py              # FastAPI
  schemas.py           # Pydantic 输入输出
  service.py           # cascade pipeline
  dashboard.py         # Streamlit/Gradio
```

接口：

- `POST /score`：单条文本风险识别。
- `POST /batch_score`：批量文本。
- `GET /entity/{name}`：主体风险画像。
- `GET /review_queue`：高风险审核队列。
- `GET /metrics`：模型服务指标。

Dashboard 页面：

- 输入金融新闻/社媒文本。
- 展示负面主体、风险类型、风险分数、证据片段。
- 展示近似 reason code。
- 展示历史同主体风险事件。

可写进简历：

> 使用 FastAPI 和 Streamlit 搭建风控审核原型系统，支持批量舆情文本评分、实体风险画像查询和高风险样本人工复核队列。

### 4.7 监控与 MLOps

风控算法实习很看重模型上线意识。建议补轻量 MLOps，不必做很重。

建议实现：

- 实验追踪：MLflow 或本地 `outputs/experiments/*.json`。
- 数据漂移：文本长度分布、实体数量分布、风险类型分布、编码器 embedding 均值漂移。
- 预测漂移：正例率、风险等级比例、LLM invalid JSON rate、幻觉实体率。
- 触发策略：漂移超过阈值后输出 `retrain_recommendation`。

可写进简历：

> 设计模型监控指标，跟踪风险正例率、实体幻觉率、预测分布漂移和数据分布漂移，支持风控模型再训练建议。

## 6. 最推荐你先做的 6 个具体任务

如果你愿意大改，我建议按这个顺序做：

1. **风险事件弱标注**：把 FINNSP 的负面主体扩展成 `risk_type/severity/evidence`。
2. **实体画像**：统计每个主体出现次数、风险类型分布、最高风险等级、典型证据。
3. **FastAPI 风控服务**：把级联推理封装成 `/score`。
4. **Dashboard**：做一个审核台，能看风险主体、风险等级、证据、历史画像。
5. **反事实评测**：构造主体替换、负面词删除、干扰主体和否定语境样本，并接入 Qwen3-8B QLoRA 做真实鲁棒性压力测试。
6. **外部泛化评测**：先跑 FLARE-zh-NSP 关键词弱监督 baseline，再接入 Qwen3-8B QLoRA 做真实外部二分类推理。

这 6 个完成后，项目就能在简历上写得比较硬了。

## 8. 面试时的讲法

建议按这个故事线讲：

1. **业务问题**：金融风险往往先在新闻和社媒爆发，风控需要知道“不只是有没有负面，而是谁有风险”。
2. **技术难点**：多主体归因、社媒噪声、LLM 幻觉、推理成本、误报/漏报代价不同。
3. **建模方案**：编码器做高吞吐初筛，Qwen3-8B QLoRA 做结构化主体复核。
4. **风控设计**：加入实体可信性、风险评分、人工审核队列。
5. **工程闭环**：脚本可复现，结果可分析，未来可 API 化和 dashboard 化。
6. **结果**：主模型 F1、Entity-F1、级联调用率、幻觉率都有可量化结果。

这套讲法比“我微调了一个 Qwen 模型”更像风控算法候选人。
