# 项目创新性改进建议

作者：10225001426 杨巨豪

## 1. 结论先行

当前项目已经具备比较完整的实验闭环：FINNSP 数据处理、TF-IDF/编码器/Qwen zero-shot/QLoRA 对比、统一 JSON 输出、实体级 F1 和图表。短板也比较明确：现在最强卖点主要是“Qwen3-8B QLoRA 指标最高”，如果论文只围绕模型分数展开，容易显得像常规课程实验，创新性不足。

建议不要继续堆更大的模型，而是把项目改成一个更贴近风控系统的研究：**面向金融舆情早筛的实体归因式风险识别框架**。具体从四个方向补创新点：

1. 加入实体候选约束和幻觉率评估，强调“负面归因必须指向文本中的金融主体”。
2. 加入风险事件类型与严重程度，避免只做二分类和主体抽取。
3. 加入编码器到 LLM 的级联风控流程，讨论准确率、实体 F1、推理成本之间的权衡。
4. 加入反事实/扰动鲁棒性实验，证明模型不是只记负面词，而是在做主体级归因。

这四点都可以在现有代码和结果上继续扩展，不需要重训一个更大的主模型。

## 2. 外部工作调研

### 2.1 FinCUGE / BBT-Fin

BBT-FinCUGE 官方应用仓库将 FinNSP 定义为“金融负面消息及其主体判定数据集”：模型需要判断文本是否包含针对某个实体的负面消息，并指出负面主体，指标为 F1-Score；原始划分为 train 4800、dev 600、test 600。该仓库同时把 FinNSP 和 FinNL、FinRE、FinFE 放在金融理解能力排行榜中，说明主流用法是做金融 NLP benchmark 和模型评测。

对本项目的启发：

- 我们不能只写“金融情感分类”，必须强调这是**负面主体判定**。
- 原始任务已经要求“主体”，因此论文应突出实体级归因指标，而不是只看二分类 F1。
- 现有开源工作偏基准评测，项目可以进一步包装为风控早筛系统。

参考：https://github.com/supersymmetry-technologies/BBT-FinCUGE-Applications

### 2.2 FinCUGE-Instruction

Hugging Face 上的 `Maciel/FinCUGE-Instruction` 是 instruction 形式的数据集，整体包含 13.8 万行，字段包括 `task`、`desc`、`instruction`、`input`、`output`，其中 FINNSP 可过滤出当前项目使用的训练/验证样本。这个版本适合做 LLM 指令微调，因为它本身已经把任务包装成中文指令。

对本项目的启发：

- 数据集本身是 instruction 数据，使用 QLoRA SFT 是合理路线。
- 论文里应说明我们进一步把自然语言输出清洗成严格 JSON schema，这是工程贡献。
- 可以增加“格式可用性”指标，例如 invalid JSON rate、fallback parse rate、schema violation rate。

参考：https://huggingface.co/datasets/Maciel/FinCUGE-Instruction

### 2.3 FLARE-zh-NSP

`ChanceFocus/flare-zh-nsp` 是一个 500 条 test split 的中文金融负面主体判定数据集，样本里包含 `choices`、`text`、`answer`、`query`、`gold` 等字段，答案为“有/无”。它没有提供主体列表级别的完整抽取标签，但非常适合做外部泛化测试。

对本项目的启发：

- 可以把它作为独立测试集，验证模型是否只在 FinCUGE eval 上好。
- 如果只评估“有/无”，就作为二分类跨数据集泛化实验。
- 如果文本中带有“实体：A;B”，可以进一步做候选实体级规则校验，但不要冒充完整人工实体标注。

参考：https://huggingface.co/datasets/ChanceFocus/flare-zh-nsp

### 2.4 FinGPT

FinGPT 强调金融场景数据变化快，重训大模型成本高，因此轻量化适配和自动数据流水线更适合金融 LLM。它的方向不是单一 FINNSP 任务，而是开放金融 LLM、instruction tuning、金融情感与预测等应用。

对本项目的启发：

- QLoRA 可以写成“低成本金融领域适配”，而不是简单模型微调。
- 可以加入“定期重训/增量更新”的风控落地讨论。
- 可以讨论“高风险样本进入 LLM 复核，普通样本由编码器处理”的成本控制。

参考：https://github.com/AI4Finance-Foundation/FinGPT

### 2.5 PIXIU / FinBen

PIXIU 和 FinBen 更像金融 LLM 综合评测框架，覆盖情绪分析、分类、知识抽取、数值理解、问答、风险管理、预测和决策等任务。FinBen 论文强调金融 LLM 不能只看单一任务，需要综合评价信息抽取、文本分析、风险管理等能力。

对本项目的启发：

- 当前项目可以把 FINNSP 扩展成“小型风控风险管理 benchmark”，不只是一个分类题。
- 风控论文里应加入部署成本、鲁棒性、实体幻觉、错误类型等系统性评估。
- 可以把“负面主体识别”上升为“金融风险事件抽取”的前置环节。

参考：https://github.com/The-FinAI/PIXIU

## 3. 当前项目的主要不足

### 3.1 创新点集中在模型替换

目前已有结果证明 Qwen3-8B QLoRA 在 FINNSP eval 上表现最好：

| 模型 | Accuracy | F1 | Entity-F1 | Invalid JSON |
|---|---:|---:|---:|---:|
| FinBERT2-large | 0.9620 | 0.9638 | 0.0000 | - |
| Qwen3-8B QLoRA | 0.9720 | 0.9732 | 0.8331 | 0.0000 |

这个结果很强，但论文如果只写“换成 Qwen3 后更好”，创新性会偏弱。因为老师可能会认为这只是模型能力提升，不是方法设计。

### 3.2 风控场景讲得还不够具体

现在已经提到“舆情早筛、风险预警、人工审核”，但还缺少可量化的风控流程，例如：

- 什么样的样本进人工复核？
- 使用 LLM 的成本如何控制？
- 错误预测会带来什么业务风险？
- 模型是否会把负面信息错归因到无辜实体？

这些问题一补，项目就更像风控算法方案。

### 3.3 实体抽取还缺少可信性约束

LLM 会输出实体，但当前只统计 Entity-F1 和 invalid JSON rate。风控场景更关心：

- 输出主体是否真的出现在原文中？
- 是否把监管机构、媒体、用户误判为风险主体？
- 多主体文本中是否能区分“被处罚者”和“监管者/报道者”？

这些指标比单纯 Entity-F1 更有业务味道。

### 3.4 错误分析和鲁棒性不足

目前论文大纲里有错误类型，但还没有形成可复现实验。建议至少自动导出 false positive、false negative、entity mismatch 三类错误样本，并增加少量反事实测试。

## 4. 建议一：实体候选约束与幻觉率评估

### 4.1 修改目标

把 LLM 输出从：

```json
{"has_negative": true, "entities": ["小资钱包", "资易贷"]}
```

升级为：

```json
{
  "has_negative": true,
  "entities": ["小资钱包", "资易贷"],
  "evidence": [
    {"entity": "小资钱包", "span": "小资钱包涉嫌诈骗"},
    {"entity": "资易贷", "span": "负责人失联"}
  ]
}
```

如果不想改训练标签，也可以先只在评估阶段增加约束指标：

- `entity_in_text_rate`：预测实体出现在原文中的比例。
- `hallucinated_entity_rate`：预测实体不在原文中的比例。
- `multi_entity_error_rate`：多主体样本中实体预测错误比例。

### 4.2 为什么有创新性

金融风控不是普通抽取任务。把风险错误归因给无风险主体，在业务上可能比漏检更严重。实体候选约束能把论文主题从“分类准确率”提升到“可信风险归因”。

### 4.3 可落地改法

- 在 `scripts/prepare_data.py` 中增加 `candidate_entities` 字段。若原始输入含“实体：A;B”，直接解析；若没有，就用训练标签实体和文本中的公司/平台关键词规则构造弱候选。
- 在 prompt 中增加硬约束：`entities` 只能从候选实体中选择；无法确定则输出空数组。
- 在 `src/risk_nlp/metrics.py` 中增加幻觉率统计。
- 论文新增表格：`Entity-F1 / entity_in_text_rate / hallucinated_entity_rate`。

优先级：最高。  
工作量：半天到一天。  
预期收益：论文创新性明显增强。

## 5. 建议二：风险事件类型与严重程度

### 5.1 修改目标

把任务从“是否负面 + 主体”扩展为“主体 + 风险类型 + 风险等级”。建议输出：

```json
{
  "has_negative": true,
  "risk_events": [
    {
      "entity": "小资钱包",
      "risk_type": "兑付/逾期风险",
      "severity": "high",
      "evidence": "到期就逾期，不回本金"
    }
  ]
}
```

### 5.2 推荐风险类型

可以定义 7 类，够风控论文使用：

| 风险类型 | 关键词例子 |
|---|---|
| 诈骗/非法集资 | 诈骗、非法吸收公众存款、集资诈骗 |
| 兑付/逾期风险 | 逾期、无法兑付、不回本金、提现困难 |
| 监管/处罚风险 | 监管提示、处罚、立案、通报 |
| 经营异常 | 失联、跑路、停业、清退、破产 |
| 财务恶化 | 净利润大幅下降、亏损、债务危机 |
| 投诉纠纷 | 投诉、维权、受害人、纠纷 |
| 洗钱/涉黑等严重违法 | 洗钱、涉黑、犯罪、刑事案件 |

### 5.3 为什么有创新性

FINNSP 原始任务只要求负面主体，但风控实际需要判断风险类型和处置优先级。加入事件类型后，论文可以讲：

- 这是从 benchmark task 到 risk event extraction 的扩展。
- 不同风险类型对应不同审核策略。
- 高风险类型可以触发人工复核或黑名单线索。

### 5.4 可落地改法

短期不必重新标完整训练集，可以做弱监督：

- 用关键词规则给训练/验证样本打 `risk_type` 和 `severity` 弱标签。
- 只在论文中把它作为扩展实验或案例分析。
- 从验证集抽 50-100 条人工快速校验，报告弱标签准确率或案例表。

优先级：高。  
工作量：一天左右。  
预期收益：风控场景讲述会更完整。

## 6. 建议三：编码器到 LLM 的级联风控系统

### 6.1 修改目标

现有结果里 FinBERT2-large 二分类 F1 已经达到 0.9638，Qwen3-8B QLoRA 二分类 F1 为 0.9732，实体 F1 为 0.8331。可以设计两阶段流程：

1. FinBERT2/MacBERT 先对全量文本做初筛。
2. 只有高风险或低置信度样本进入 Qwen3-8B QLoRA。
3. LLM 输出结构化主体，供人工审核或风险画像系统使用。

### 6.2 建议实验

模拟三个阈值策略：

| 策略 | LLM 调用比例 | 用途 |
|---|---:|---|
| Conservative | 20%-30% | 只复核最高风险和最不确定样本 |
| Balanced | 40%-60% | 兼顾准确率和成本 |
| Recall-first | 70%-80% | 风控宁可多报，减少漏报 |

论文报告：

- Overall F1
- Recall
- Entity-F1
- LLM call rate
- 平均推理耗时估算

### 6.3 为什么有创新性

这比单模型排行榜更像真实风控系统。老师看到的不只是“我训练了大模型”，而是“我设计了可部署的成本敏感风控 pipeline”。

优先级：最高。  
工作量：半天到一天。  
预期收益：论文风控味最强。

## 7. 建议四：反事实与扰动鲁棒性评估

### 7.1 修改目标

构造小规模反事实测试集，验证模型是否真正理解主体归因：

| 原样本 | 扰动方式 | 期望 |
|---|---|---|
| A 平台涉嫌诈骗，B 银行回应 | 交换 A/B 位置 | 仍识别 A |
| A 平台涉嫌诈骗 | 删除“涉嫌诈骗” | 变为无负面 |
| A 平台被监管处罚 | 把 A 替换为同类主体 C | 识别 C |
| A 出现，B 被处罚 | 增加无关主体 C | 不输出 C |

### 7.2 指标

- `counterfactual_consistency`：扰动后是否符合预期。
- `entity_switch_accuracy`：实体替换后是否能同步替换预测主体。
- `negative_cue_sensitivity`：删除负面触发词后是否从有变无。

### 7.3 为什么有创新性

很多模型在 FINNSP 上高分，可能只是记住“诈骗/逾期/处罚”等负面词。反事实测试能证明模型不是简单关键词分类，而是在做实体和事件之间的归因。

优先级：高。  
工作量：半天。  
预期收益：错误分析和方法可信度明显增强。

## 8. 建议五：外部泛化测试

### 8.1 修改目标

使用 `ChanceFocus/flare-zh-nsp` 的 500 条 test 样本做外部验证。因为它答案是“有/无”，可以先只评估二分类泛化：

- Qwen3-8B QLoRA 在 FINNSP eval 上的二分类指标。
- 同一模型在 FLARE-zh-NSP 上的二分类指标。
- 分析域偏移：文本风格、实体格式、社媒噪声。

### 8.2 注意事项

不要把 FLARE-zh-NSP 当作完整实体抽取测试集，因为它主要给的是“有/无”。如果想做实体层面，只能把文本中的“实体：A;B”作为候选集合，做辅助分析。

优先级：中。  
工作量：半天到一天。  
预期收益：增强论文严谨性。

## 9. 建议六：错误案例自动导出

### 9.1 修改目标

从主模型验证集预测文件自动导出：

- `false_positive.jsonl`
- `false_negative.jsonl`
- `entity_mismatch.jsonl`
- `multi_entity_cases.jsonl`

每条保留：

```json
{
  "id": "eval-xxx",
  "text": "...",
  "gold": {"has_negative": true, "entities": ["A"]},
  "pred": {"has_negative": true, "entities": ["B"]},
  "error_type": "entity_mismatch"
}
```

### 9.2 论文写法

错误分析不要只写“模型有时会错”，而要写成风控风险：

- False Negative：高风险事件漏报，影响预警召回。
- False Positive：增加人工审核成本。
- Entity Mismatch：风险归因错误，可能误伤无风险主体。
- Hallucinated Entity：LLM 幻觉，必须上线前规则拦截。

优先级：最高。  
工作量：半天。  
预期收益：论文结果分析更扎实。

## 10. 推荐实施顺序

### 第一优先级：马上值得做

1. 错误案例自动导出。
2. 实体幻觉率和实体在文率统计。
3. 编码器到 LLM 级联风控模拟。
4. 反事实测试 30-50 条。

这四项最适合补“创新性不足”的问题，而且不需要重训主模型。

### 第二优先级：时间允许再做

1. 风险类型/严重程度弱标签。
2. FLARE-zh-NSP 外部泛化测试。
3. prompt 加候选实体约束后，重新评估 Qwen3-8B QLoRA。

### 第三优先级：不建议现在做

1. 继续训练更大的 9B/14B 模型。
2. 大规模重新标注训练集。
3. 追求小数点级别的 Accuracy 提升。

当前主模型已经能拿到 0.9720 Accuracy 和 0.8331 Entity-F1，继续堆模型的边际收益不如补评测和业务闭环。

## 11. 论文中可以这样包装贡献

建议把论文贡献改成下面四条：

1. 本文构建了一个统一 JSON schema 的中文金融负面主体识别框架，同时评估二分类和实体级风险归因。
2. 本文基于 Qwen3-8B 进行 QLoRA 指令微调，在 FINNSP 验证集上取得 0.9732 F1 和 0.8331 Entity-F1，优于传统机器学习、通用中文编码器和金融领域编码器基线。
3. 本文提出面向风控早筛的两阶段级联流程，用轻量编码器进行全量初筛，用 LLM 对高风险或不确定样本进行主体级复核，从而兼顾准确率、召回率和推理成本。
4. 本文从实体幻觉率、错误类型、反事实扰动和外部泛化等角度分析模型可靠性，强调金融风控场景中“负面归因正确性”比普通情感分类更重要。

## 12. 最小改动版本

如果时间非常紧，至少做下面三件事：

1. 新增错误案例导出脚本和错误分析表。
2. 新增实体在文率/幻觉率指标。
3. 在论文中加入级联风控流程图和成本讨论。

这三项能最快把项目从“模型对比实验”抬到“风控算法系统方案”。
